# ET UpCloud — Infrastructure Skills

Deploy and manage full-stack applications on UpCloud using the `upctl` CLI, Docker Compose, Caddy, and secure secret management.

## Quick Reference

| Item            | Value                                    |
| --------------- | ---------------------------------------- |
| CLI             | `upctl` (UpCloud CLI)                    |
| Default Zone    | `fi-hel1` (Helsinki, EU)                 |
| Config File     | `.deploy.json` (per-project, git-safe)   |
| Secrets         | Infisical / Docker Secrets / S3 Bundle (low-risk only) |
| Reverse Proxy   | Caddy (auto-SSL)                         |
| Container       | Docker Compose                           |
| DB              | Managed PostgreSQL + pgvector/pg_cron    |
| Object Storage  | S3-compatible UpCloud Object Storage     |

## Skills

- **`/upcloud:start`** — Interactive onboarding wizard (discover → recommend → plan → provision → generate scripts → onboard)
- **`/upcloud:setup`** — Direct infrastructure provisioning (for users who know what they want)
- **`/upcloud:deploy`** — Continuous deployment (push, migrate, up, status, logs, rollback, secrets)

## Key Files

- `.deploy.json` — Project config generated by setup (UUIDs, endpoints, zone). Checked into git, no secrets.
- `docker-compose.prod.yml` — Production compose file with Caddy + app services
- `Caddyfile` — Auto-SSL reverse proxy configuration

## Secret Management — Low Risk Only

Only three providers are supported, all rated low or acceptable risk:

| Provider | Risk | Rotation | Audit Trail | Best for |
| -------- | ---- | -------- | ----------- | -------- |
| **Infisical** (self-hosted) | Low | Automatic | Full | Production, compliance |
| **Docker Secrets** | Low | Manual | None | Simple/internal apps |
| **S3 Bundle** (encrypted) | Acceptable | Manual | S3 logs | Teams with S3 tooling |

**Encrypted .env in containers is never an option** — high risk, no rotation, no audit trail.

The chosen provider is stored in `.deploy.json` under `secrets.provider`.

## Pre-flight Checks

**Always run before creating a PR or deploying:**

```bash
claude plugin validate .
claude plugin validate ./et-upcloud-plugin
```

Both must pass. Do not push, create a PR, or deploy if validation fails.

## Deletion Policy — MANDATORY USER APPROVAL

**Every destructive operation on UpCloud requires explicit user approval. No exceptions.**

Enforced via `settings.json` deny list — all `upctl ... delete` and `upctl ip-address remove` commands are blocked and will prompt the user. This also applies to:
- `docker compose down`, `docker image prune`, `docker system prune` via SSH
- `DROP TABLE`, `DROP DATABASE`, `DROP ROLE` via SQL
- Deleting secret files or Infisical secrets

**Never automate, script around, or batch delete operations. Each deletion must be individually confirmed by the user.**

## Principles

1. **No secrets on disk** — Secrets flow through the configured provider. Never in .env files, never in images.
2. **No secrets in images** — Injected at runtime (Infisical CLI, Docker tmpfs mount, or entrypoint fetch).
3. **No deletes without user approval** — Enforced via permission deny list. Every destructive operation prompts the user.
4. **EU data residency** — Default zones: `fi-hel1`, `de-fra1`.
5. **Least privilege** — Three DB roles (`app_rw`, `app_ro`, `app_migrate`). Scoped storage keys. Machine identities for automation.
6. **Idempotent** — Running setup twice skips existing resources.
7. **Observable** — Every deploy shows status, health check, logs.
